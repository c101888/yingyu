# 开发规则与测试验收标准

## 核心原则

**每次修改后必须做测试和验收，未经验证不得声明修复完成。**

---

## 1. 测试验收规则（必须遵守）

### 1.1 后端 API 修改
- 使用 `curl` 或 `Invoke-WebRequest` 测试相关 API 端点
- 验证返回状态码和响应体
- 测试正常流程和异常流程（如无效 token、超限等）
- 记录测试命令和结果

### 1.2 前端修改
- 本地启动 dev server（`npm run dev`）验证页面渲染
- 检查浏览器控制台无报错
- 测试手机/平板/桌面三种视口尺寸
- 验证交互功能（点击、输入、跳转）

### 1.3 数据库修改
- 在本地数据库测试 SQL 语句
- 验证数据完整性和约束
- 如涉及生产数据，先备份再修改

### 1.4 部署后验收
- 部署到服务器后，必须通过 HTTP 请求验证线上功能
- 检查静态文件是否更新（对比 JS 文件名 hash）
- 验证 Cache-Control 头是否生效
- 清除浏览器缓存后重新测试

### 1.5 UI 布局修改
- 在浏览器开发者工具中切换设备模拟器
- 测试 iPhone SE（375px）、iPad（768px）、Desktop（1280px）三种尺寸
- 确认无溢出、无换行、无空白浪费
- 截图对比修改前后效果

---

## 2. UI 布局设计规范

### 2.1 响应式断点
- 手机：`< 640px`（默认样式，不带 sm: 前缀）
- 平板：`640px - 1024px`（使用 `sm:` 前缀）
- 桌面：`> 1024px`（使用 `lg:` 前缀）

### 2.1.1 ⚠️ 响应式样式隔离规则（核心红线）
**手机版修改绝对不能影响 web/平板版样式。** 这是最高优先级规则。

1. **修改手机版样式时，必须加 `sm:` 前缀**：所有只针对手机的样式（padding、gap、font-size、height 等）必须使用无前缀类（如 `p-4`），对应的 web/平板版样式必须用 `sm:` 前缀覆盖（如 `sm:p-5`）。
2. **禁止使用无前缀类直接修改全局样式**：例如把 `p-5` 改成 `p-4` 会同时影响所有屏幕尺寸。正确做法是 `p-4 sm:p-5`（手机 p-4，640px+ 恢复 p-5）。
3. **CardContent padding 统一模式**：所有卡片内容区使用 `p-4 sm:p-5` 模式（手机 16px，web 20px）。特殊紧凑卡片可用 `p-2.5 sm:p-5`。禁止混用 `p-5`、`p-6` 无前缀。
4. **修改前先对比 git 历史**：如不确定原始值，用 `git diff 61bfd84 HEAD -- <file>` 对比手机优化前的版本。
5. **三端独立验证**：修改后必须在手机（375px）、平板（768px）、桌面（1280px）三种视口下检查 padding/gap 一致性。

### 2.2 网格布局规则
- **选项/卡片类**：手机 2 列、平板 3 列、桌面 3-4 列
- **列表项**：手机 1 列、平板 2 列、桌面 2-3 列
- **按钮组**：手机可换行，平板/桌面一行

### 2.3 移动端禁止的布局
- 单个选项独占一行（除非内容确实很长）
- 高度过大的卡片（移动端高度应紧凑）
- 文字换行显示（应调整布局或字号）
- 固定宽度的元素导致横向滚动

### 2.4 移动端触摸友好
- 按钮最小 44x44px
- 间距至少 8px
- 避免过小的点击区域

## 3. 代码质量规则

### 3.1 错误处理
- 所有 API 调用必须有 try/catch
- 401 错误使用 `isAuthError(err)` 静默处理
- 用户可见错误用 toast/alert 显示，不阻塞界面

### 3.2 状态管理
- zustand store 的 persist 配置必须用 `merge` 函数覆盖旧值
- 不持久化会变更的配置常量（如 guestMaxUsage）

### 3.3 部署检查清单
- [ ] `npm run build` 成功
- [ ] 上传 dist 到服务器
- [ ] 验证服务器 JS 文件已更新
- [ ] 验证 Cache-Control 头
- [ ] 测试核心功能（登录、生成、tier）
- [ ] 清除浏览器缓存测试

## 4. 常见问题排查

### 4.1 "登录已过期"
1. 检查服务器数据库是否有该账号
2. 检查密码是否正确
3. 检查 token 是否过期
4. 检查前端是否清除了旧 token

### 4.2 静态文件不更新
1. 检查 nginx `Cache-Control` 头
2. 检查 JS 文件名 hash 是否变化
3. 强制刷新浏览器（Ctrl+F5）

### 4.3 API 返回 401
1. 检查 Authorization 头是否正确
2. 检查 token 是否有效
3. 检查账号是否存在于数据库

## 5. Git 本地仓库备份制度

### 5.1 自动备份规则
**每次用户指令修改完成后，必须自动提交 git 备份，便于随时回滚版本。**

1. **提交时机**：每完成一轮修改（一个用户指令的所有改动），立即提交。
2. **提交命令**（PowerShell，用 `;` 分隔，不能用 `&&`）：
   ```powershell
   git add -A; git commit -m "描述本次修改内容"
   ```
3. **提交信息规范**：简明描述修改的模块和目的，例如：
   - `修复: Profile/SceneResult/Upgrade 等页面 padding 响应式隔离`
   - `功能: Quiz 选项动态列数适配`
   - `构建: 部署最新版本到服务器`
4. **不要 push**：除非用户明确要求，只做本地提交。

### 5.2 版本回滚
- 查看历史：`git log --oneline -10`
- 对比差异：`git diff <旧commit> HEAD -- <file>`
- 回滚某文件：`git checkout <旧commit> -- <file>`
- 回滚整个版本：`git reset --hard <commit>`（谨慎使用）

### 5.3 关键版本节点
- `0e11b00`：初始完整版 MVP
- `61bfd84`：手机优化前的版本（响应式 UI 参考基准）
- 后续每次修改后的 commit 均为可回滚点
